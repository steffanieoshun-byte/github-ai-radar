# 下一阶段可开发内容

> 状态：暂存为复盘材料，暂时不进入开发。

## 当前基线

- 本地 Web 应用已经跑通：关键词扫描、GitHub 数据读取、SQLite 保存、历史项目维护、项目详情展示。
- 已接入 GitHub token 和 DeepSeek API，当前核心分析不再依赖 Mock。
- 产品方向已经明确：不是排行榜，不是测评站，而是一个帮助发现 AI 工具、治理方法、知识技巧和可复刻灵感的本地雷达。
- 当前最重要的质量标准是：每条结果必须让用户知道“它到底是什么、为什么值得看、能带来什么灵感、最小可复刻动作是什么”。

## 下一阶段优先级

### P0：统一内容质量

目标：解决旧数据和早期模板分析导致的内容质量不一致。

可做内容：

- 增加“批量重分析”功能，把旧 Mock 记录升级为真实 LLM 分析。
- 在项目详情中显示“分析来源”和“是否需要重分析”的清晰状态。
- 对低质量分析增加重新生成入口，而不是让用户手动删除后重扫。

验收边界：

- 老记录不会被覆盖，新的分析作为新历史保存。
- 用户能看出某条内容是 Mock、模型失败回退，还是 DeepSeek 分析。
- 批量重分析失败时不中断整个应用，并保留失败原因。

### P1：优化分析提示词和判断流

目标：让分析结果更像“有判断的研究助理”，少像“字段填表”。

可做内容：

- 重写 LLM prompt，强制输出四类判断：
  - 它到底是什么
  - 为什么留下或为什么跳过
  - 对我有什么灵感
  - 最小可复刻动作是什么
- 要求模型引用 README、目录、关键文件或 issue 等证据，证据不足时明确写“证据不足”。
- 对不同搜索意图使用不同判断标准，例如 AI 工具、AI 治理、知识技巧、量化、自动化工作流。

验收边界：

- 同一关键词下，不同仓库的结论明显不同。
- 结论不能只写“可能有用”，必须指出具体有用在哪里。
- 低证据项目不会被强行说成高价值项目。

### P1：后台维护库

目标：把“删除、跳过、观察、实验候选”从隐性状态变成可维护资产。

可做内容：

- 增加一个后台维护页，展示：
  - 已删除项目
  - 已跳过项目
  - 已观察项目
  - 实验候选项目
- 支持恢复删除项目。
- 支持查看某个项目为什么被跳过，避免重复扫描。

验收边界：

- 前台删除只是隐藏，不会删除数据库记录。
- 重新扫描时能识别已扫过、已跳过、已删除项目。
- 用户可以从后台恢复误删项目。

### P2：多模型配置和回退

目标：让用户可以根据额度、速度和质量切换模型。

可做内容：

- 支持 OpenAI-compatible API 的主模型和备用模型。
- 增加模型健康检查，显示当前是否能正常调用。
- 增加失败回退策略：主模型失败后自动尝试备用模型，再失败才回 Mock。

验收边界：

- 不要求用户把 key 贴到聊天窗口。
- `.env.example` 说明清楚每个模型配置项。
- 页面能显示本次分析实际使用了哪个模型。

### P2：搜索策略升级

目标：让“搜索命令标准”更聪明，减少噪音。

可做内容：

- 为不同搜索意图维护搜索词扩展规则。
- 对数量不足的情况给出解释：是 GitHub 搜索少，还是初筛过滤多。
- 增加关键词历史和常用方向快捷选择。

验收边界：

- 用户输入数量 10，结果不足 10 时，页面能解释原因。
- 数量 1 不会因为去重直接空白。
- PASS 和去重机制不会误杀新关键词下的有效项目。

### P3：灵感库管理体验

目标：让用户长期维护这个库，而不是只看一次扫描结果。

可做内容：

- 增加时间排序、关键词筛选、标签筛选。
- 增加项目备注字段，用户可以写自己的判断。
- 增加“今天值得看”或“最近高灵感”视图。

验收边界：

- 左侧列表能承载大量历史项目，不显得难管理。
- 用户不需要理解扫描过程，也能维护自己的灵感库。
- 备注和状态更新不会覆盖分析历史。

## 暂时不做

- 不做公开排行榜。
- 不做账号登录。
- 不做云端部署。
- 不做大规模爬虫。
- 不自动 clone 仓库。
- 不做复杂前端框架迁移。
- 不把本地 SQLite 改成远程数据库。
